# .gitignore Configuration Explained

This document explains each section of the `.gitignore` file in the Vxture project, focusing on the current monorepo, tech stack, and best practices for modern web development.
本文档详细解释了 Vxture 项目中 `.gitignore` 文件的各个部分及其作用，聚焦于当前单体仓库、技术栈和现代 Web 开发的最佳实践。

## What is .gitignore?

The `.gitignore` file tells Git which untracked files to ignore. These are usually files generated locally that should not be committed to the repository, such as build outputs, dependencies, local configs, and logs.

## Section Details

### Dependencies

```ignore
# dependencies
/node_modules
/.pnp
.pnp.js
```

| Item            | Description                                                         |
| --------------- | ------------------------------------------------------------------- |
| `/node_modules` | All npm dependencies, large and can be restored from `package.json` |
| `/.pnp`         | Yarn Plug'n'Play dependency cache                                   |
| `.pnp.js`       | Yarn Plug'n'Play config file                                        |

These should not be versioned because:

1. They are large and bloat the repo
2. They can be regenerated with `npm install` or `yarn install`
3. Platform-specific binaries may differ (Windows/Linux)

### Testing

```ignore
# testing
/coverage
```

| Item        | Description                               |
| ----------- | ----------------------------------------- |
| `/coverage` | Test coverage reports, generated by tools |

Coverage reports are local and should not be versioned.

### Next.js Build Files

```ignore
# next.js
/.next/
/out/
```

| Item      | Description                                  |
| --------- | -------------------------------------------- |
| `/.next/` | Next.js build and cache files (dev and prod) |
| `/out/`   | Static export output from `next export`      |

These contain compiled code and assets, and should be regenerated per environment.

### Production Build Files

```ignore
# production
/build
```

| Item     | Description                       |
| -------- | --------------------------------- |
| `/build` | Production build output directory |

Should be generated during deployment, not versioned.

### System & Temp Files

```ignore
# misc
.DS_Store
*.pem
```

| Item        | Description                              |
| ----------- | ---------------------------------------- |
| `.DS_Store` | macOS metadata file, auto-generated      |
| `*.pem`     | Private key/certificate files, sensitive |

These are either system files or sensitive and should not be versioned.

### Debug Logs

```ignore

npm-debug.log*
yarn-debug.log*
yarn-error.log*
```

| Item              | Description          |
| ----------------- | -------------------- |
| `npm-debug.log*`  | npm error log files  |
| `yarn-debug.log*` | Yarn debug log files |
| `yarn-error.log*` | Yarn error log files |

Debug logs are local and should not be versioned.

### Environment Variable Files

| Item          | Description                                                         |
| ------------- | ------------------------------------------------------------------- |
| `.env*.local` | Local env files (e.g. `.env.local`, `.env.development.local`, etc.) |

| Item      | Description                          |
| --------- | ------------------------------------ |

| --------------- | ------------------------------------------- |

These are auto-generated and can be safely ignored.

## Best Practices

1. **Global vs Project .gitignore**:
   - Use a global `.gitignore` for OS/editor files (e.g. `.DS_Store`)
   - Use project `.gitignore` (this file) for project-specific rules

2. **Sensitive Info Templates**:
   - Provide a template (e.g. `.env.example`) for env files
   - The template lists required variables but no real secrets

3. **Pattern Rules**:
   - `*` matches any chars except `/`
   - `**` matches any directory depth
   - `?` matches a single char
   - `[abc]` matches a character set
   - `!` negates a rule (includes a previously ignored file)

4. **IDE/Editor Config**:
   - Add IDE-specific files (e.g. `.idea/`) to your global `.gitignore`
   - Consider versioning shared IDE config for team consistency

5. **Un-ignore Specific Files**:
   - Use `!` to include files that would otherwise be ignored
   - Example: `!important.log` includes `important.log` even if `*.log` is ignored
