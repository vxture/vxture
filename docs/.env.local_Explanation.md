
# Environment Variable Configuration Explained

This document explains how environment variables are managed in the Vxture project, including file conventions, variable meanings, and best practices for secure and effective configuration.
本文档说明了 Vxture 项目中环境变量的管理方式，包括文件约定、变量含义及安全高效配置的最佳实践。

## Overview of Environment Variable Files

Next.js projects typically use `.env.*` files to manage environment variables. In Vxture, the following files are used:

- `.env.local.template` – Template listing all required variables and descriptions
- `.env.local` – Local development environment variables (not committed to version control)

## .env.local.template Example

```dotenv
# Development environment variables

# App settings
APP_ENV=development
DEBUG=true
SECRET_KEY=your-secret-key-here

# Database
DATABASE_URL=postgresql://postgres:password@localhost:5432/vxture
REDIS_URL=redis://localhost:6379/0

# AI Model APIs
# DashScope (Alibaba)
DASHSCOPE_API_KEY=your-dashscope-api-key

# DeepSeek
DEEPSEEK_API_KEY=your-deepseek-api-key

# Optional: OpenAI (if used)
# OPENAI_API_KEY=your-openai-api-key
# OPENAI_API_BASE=https://api.openai.com/v1
```

### Variable Details

#### App Settings

| Variable      | Example Value            | Description                                                                 |
|-------------- |-------------------------|-----------------------------------------------------------------------------|
| `APP_ENV`     | `development`           | App environment: `development`, `testing`, or `production`                  |
| `DEBUG`       | `true`                  | Enable debug mode (should be `false` in production)                         |
| `SECRET_KEY`  | `your-secret-key-here`  | Secret key for encryption, sessions, tokens; use a strong random value      |

#### Database

| Variable         | Example Value                                             | Description                                                                |
|------------------|----------------------------------------------------------|----------------------------------------------------------------------------|
| `DATABASE_URL`   | `postgresql://postgres:password@localhost:5432/vxture`    | PostgreSQL connection string                                               |
| `REDIS_URL`      | `redis://localhost:6379/0`                                | Redis connection string for cache/session                                  |

#### AI Model APIs

| Variable              | Example Value                | Description                                         |
|-----------------------|-----------------------------|-----------------------------------------------------|
| `DASHSCOPE_API_KEY`   | `your-dashscope-api-key`     | DashScope (Alibaba) API key                         |
| `DEEPSEEK_API_KEY`    | `your-deepseek-api-key`      | DeepSeek API key                                    |
| `OPENAI_API_KEY`      | `your-openai-api-key`        | (Optional) OpenAI API key                           |
| `OPENAI_API_BASE`     | `https://api.openai.com/v1`  | (Optional) OpenAI API base URL (custom endpoint)    |

## Usage Guide

### Setting Up Local Environment Variables

1. Copy the template to create `.env.local`:
   ```bash
   cp .env.local.template .env.local
   ```
2. Edit `.env.local` and fill in actual values.
3. Restart the dev server to apply changes:
   ```bash
   npm run dev
   ```

### Accessing Environment Variables in Code

#### Frontend (Browser)

Only variables prefixed with `NEXT_PUBLIC_` are exposed to the browser:

```typescript
// This will be replaced at build time
const apiUrl = process.env.NEXT_PUBLIC_API_URL;
```

#### Server-side

All variables are accessible in server components, API routes, and server actions:

```typescript
const secretKey = process.env.SECRET_KEY;
const dbUrl = process.env.DATABASE_URL;
```

### Production Configuration

In production, use your deployment platform's environment variable settings (e.g. Vercel, Railway) instead of `.env` files.

## Best Practices

1. **Security**
   - Never commit `.env.local` with real secrets to version control
   - Use strong, random values for secrets and passwords
   - Rotate secrets and passwords regularly
2. **Naming Conventions**
   - Use uppercase and underscores (e.g. `DATABASE_URL`)
   - Only use `NEXT_PUBLIC_` prefix for variables needed in the browser
3. **Type Safety**
   - Use a library like `zod` to validate environment variables:

   ```typescript
   // src/lib/env.ts
   import { z } from 'zod';

   const envSchema = z.object({
     DATABASE_URL: z.string().url(),
     REDIS_URL: z.string().url(),
     SECRET_KEY: z.string().min(32),
     // ...other variables
   });

   export const env = envSchema.parse(process.env);
   ```
4. **Environment-specific Files**
   - Development: `.env.development.local`
   - Testing: `.env.test.local`
   - Production: set via deployment platform
5. **Debugging**
   - Check file names (`.env.local`)
   - Check variable names (case-sensitive)
   - Restart the dev server after changes
