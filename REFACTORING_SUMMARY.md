## 💡 为什么这样设计？

### 问题：为什么要分离数据？

**答**：让内容管理与代码管理分离，各司其职。

```
当前（紧耦合）:
产品经理 → 给设计师 → 设计师告诉开发者 → 开发者改代码 → 测试 → 发版
这个流程最短 1 天，通常 1 周。

改造后（解耦）:
产品经理 → 直接在 Strapi 更新 → 内容立即生效
这个流程是秒级的。
```

### 问题：为什么要升级 i18n？

**答**：Zustand Store 不是翻译管理工具，i18next 才是行业标准。

```
Zustand (当前):
├─ 优: 轻量
└─ 缺: 没有复数、插值、动态加载等特性

i18next (改造后):
├─ 优: 业界标准，支持复数、插值、命名空间、动态加载
├─ 优: 翻译缺失提醒
├─ 优: 可集成翻译管理平台（如 Crowdin）
└─ 缺: 多一个依赖（可接受）
```

### 问题：为什么要支持 System 主题？

**答**：用户体验，尊重用户的系统偏好。

```
用户系统设置为深色 → 用户访问网站 → 网站显示浅色 → 刺眼
改造后 → 网站自动显示深色 → 用户满意

这是现代 Web 应用的基本做法。
```

### 问题：为什么要这样设计 API？

**答**：降级策略保证可靠性。

```
想象你的服务器宕机了：

改造前:
└─ 网站完全无法使用 😞

改造后:
├─ API 不可用？降级到 JSON
├─ JSON 加载失败？降级到内存默认数据
└─ 至少能展示基本内容 🙂
```

## 🗓️ 实施时间表

### Week 1: 基础改造 (数据 + 翻译 + 主题)

**投入**: 2-3 天
**收益**: 数据分离、系统主题检测

```
Day 1:
  ✅ 创建 public/data/ 和 src/locales/
  ✅ 导出组件中的硬编码内容到 JSON
  ✅ 导出翻译到 src/locales/*.json
  ✅ Git commit

Day 2:
  ✅ 创建 contentService.ts
  ✅ 创建 useContent() Hook
  ✅ 更新 3-5 个主要组件使用 useContent()
  ✅ 测试

Day 3:
  ✅ 升级主题 Store 支持 'system'
  ✅ 添加系统主题检测和媒体查询监听
  ✅ 更新 Header 主题选择器
  ✅ 完整测试
```

**交付物**:

- `public/data/*.json` (所有内容)
- `src/locales/` (所有翻译)
- `src/services/contentService.ts` (内容加载)
- 更新的组件 (使用 useContent)
- 升级的主题系统

**验证标准**:

- ✅ `pnpm type-check` 通过
- ✅ 内容能正常加载和显示
- ✅ 主题切换正常（包括 System）
- ✅ 语言切换正常
- ✅ 没有控制台错误

### Week 2: i18next 集成 (可选)

**投入**: 1-2 天
**收益**: 标准化翻译管理

```
Day 1:
  ✅ 安装 i18next 依赖

Day 2:
  ✅ 迁移 2-3 个关键组件使用 useTranslation()
```

**交付物**:

- `i18next.config.ts` (i18next 配置)
- 迁移的组件示例
- 迁移指南
